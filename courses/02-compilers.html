Module 0 · 20 min

What a compiler actually is

A compiler is a semantics-preserving translator: it turns a program in language A into a program in language B such that both mean the same thing. Everything else — parsing, optimizing, register allocation — is implementation detail in service of that one contract.

Three consequences worth internalizing on day one:

  • "Mean the same thing" requires a definition of meaning. Languages have specifications (semantics). Where the spec is silent — undefined behavior — the compiler may do anything. No spec, no compiler; just a converter that works until it doesn't. (Course 3 will ask: what is the spec of a geometry kernel? Answer: there isn't one. Hold that thought.)
  • Translation is directional. Lowering (high-level → low-level) discards structure by design: your loops, types, and variable names die so the machine can eat. Raising (low → high, i.e. decompilation) means reconstructing discarded intent — guesswork, not lookup. Lowering is a function; lifting is search.
  • The pipeline shape is universal. Front end (understand the source) → middle end (improve a neutral representation) → back end (emit the target). Every compiler you'll ever meet — GCC, LLVM, TVM, the 400-line one in this repo — is this shape.

Module 1 · 45 min

The front end: text → tree

Three stages, all mechanical:

  1. Lexing: characters → tokens. "wall_len - win_w" becomes [ID(wall_len), MINUS, ID(win_w)]. Regular expressions suffice.
  2. Parsing: tokens → tree, according to a grammar (precedence: * binds tighter than +, parentheses group). Output is an AST (abstract syntax tree): Sub(Ref("wall_len"), Ref("win_w")). Recursive descent — a function per grammar rule — is the technique you can write by hand in an afternoon.
  3. Semantic analysis: walk the tree, resolve names, check types ("you can't add a solid to a number"), reject nonsense early.
You already own a working example. This repo's geomir/scad.py contains a complete hand-written front end for a subset of OpenSCAD: _TOKEN regex = lexer; the _Lifter.expr/term/factor methods = a precedence-climbing recursive-descent parser; geomir/ir.py's verify() = semantic analysis (SSA def-before-use, operand kinds, export integrity). Read those ~200 lines after this module — front ends will permanently demystify.

Module 2 · 60 min

IRs and SSA: the compiler's native language

ASTs mirror source syntax — awkward for optimization. So compilers translate into an intermediate representation (IR): a language designed for analysis, not for humans. The dominant design since the 1990s is SSA — static single assignment (Cytron et al., IBM, 1991): every value is assigned exactly once, and every use names its unique definition.

// source            // SSA form
x = a + b;            %x1 = add %a, %b
x = x * 2;            %x2 = mul %x1, 2
y = x + 1;            %y1 = add %x2, 1

Why this is a superpower: def-use chains become explicit graph edges. "Where did this value come from?" is a pointer lookup, not a dataflow analysis. Dead code = values with no uses. Constant propagation, common-subexpression elimination, dependency analysis — all become graph walks. (Control flow joins need φ "phi" functions to merge values from different paths — a detail you can absorb later; straight-line SSA is the concept that matters today.)

You already own a working example, again. The recipe IR in this repo is SSA: every %name in recipes/studio_wall.ir is defined once (the verifier rejects redefinition) and the evaluator resolves uses by name. The "backward slice" in geomir/eval.py (mark()) — walking from an export back through its transitive dependencies — is textbook SSA def-use traversal, and is what makes per-element evaluation possible.

Two more IR concepts you'll meet constantly:

  • Multiple IRs per compiler. Real pipelines lower through several: e.g. AST → high-level IR → SSA IR → machine IR. Each level makes some class of transformation natural and others impossible. Which level you standardize on determines what you can express — the single most important sentence for Course 3.
  • Textual + in-memory forms. Serious IRs have a printable, re-parsable text format (LLVM .ll, MLIR .mlir, our .ir) — essential for testing (golden files, round-trip checks: see print/parse fixed point in this repo's test suite).

Module 3 · 60 min

The middle end: optimization & e-graphs

The middle end is a sequence of passes: each takes IR, returns better IR, preserving semantics. Classics:

PassWhat it does
Constant folding2 * 8 → 16 at compile time
Dead code eliminationdelete values nothing uses (trivial in SSA)
Common subexpression eliminationcompute (a+b) once, reuse it
Inliningreplace a call with the callee's body — the gateway drug: exposes all other optimizations
Canonicalizationrewrite equivalent forms to one normal form (x*2 → x<<1) so later passes match fewer patterns
Loop transformsunroll, fuse, tile, vectorize — where the big performance lives

The phase-ordering problem: passes interact — inlining before folding beats the reverse; canonicalizing one way helps pass A, hurts pass B. Pass order is a dark art of pipeline tuning.

E-graphs: don't pick an order, keep everything

The modern answer to phase-ordering is equality saturation (Tate et al. 2009; the egg library, POPL 2021): instead of destructively rewriting, an e-graph compactly stores all equivalent forms simultaneously. Apply rewrite rules until saturation, then extract the best form by any cost function you like. No ordering problem — every form coexists.

Remember this for Course 3: Szalinski (PLDI 2020) uses e-graphs on CAD programs — many equivalent constructions of the same shape held at once, then extract the most editable one. "Same shape, different recipe" is exactly the cross-CAD translation problem.

Module 4 · 45 min

The back end: IR → machine, linking, debug info

  • Instruction selection: map IR ops onto the target's instruction set (ISA) — pattern matching IR trees against instructions.
  • Register allocation: infinite IR values → 16-ish physical registers (graph coloring); spill the rest to memory.
  • Scheduling & emission: order instructions for the pipeline, emit object code.
  • Linking: combine object files, resolve symbols. Symbols matter: functions/data are referenced by stable names that survive recompilation of other modules — the reason separate compilation works at all. (Course 1's persistent-naming problem is exactly a symbol-stability failure.)
  • Debug info (DWARF): compilers emit side-tables mapping machine code back to source lines and variables. The binary carries provenance to its own source — lowering plus a paper trail. (Course 3 steals this idea outright: geometry artifacts that carry links back to their recipe.)

Also in the back-end family: runtimes (allocators, exception tables), ABIs (call conventions — the "tolerances" of the software world: mismatch them and two correct modules corrupt each other), and library calls — the compiler doesn't generate everything; it links against hand-optimized vendor code (BLAS, libm). Keep that one too; Relax makes it first-class.

Module 5 · 45 min

LLVM: why one IR conquered the world

LLVM (Chris Lattner, UIUC, ~2000–03; supercharged at Apple from 2005; Clang C/C++ front end 2007) restructured the industry around one idea: a typed, SSA-based, target-independent IR as the universal middle layer.

define i32 @square(i32 %x) {
entry:
  %y = mul nsw i32 %x, %x
  ret i32 %y
}

Any language that lowers to LLVM IR (C, C++, Rust, Swift, Julia, Zig…) instantly gets every optimization pass and every backend (x86, ARM, RISC-V, GPUs, WebAssembly). M front ends + N back ends = M+N components instead of M×N compilers. Why it worked — three preconditions worth memorizing, because Course 3 tests whether they hold for geometry:

  1. A shared abstract machine. All those languages ultimately target the same model of computation — linear memory, integer/float arithmetic with defined semantics. There was a common "bottom" to lower to.
  2. Specified semantics. LLVM IR has a reference manual; every op's meaning is documented, including the undefined-behavior contract that licenses optimization.
  3. Aligned incentives. Apple, Google, and everyone else needed it and funded it; its permissive license made adoption free. Nobody's moat depended on the IR being bad.

What LLVM is bad at (setting up the next module): LLVM IR is one level — scalar ops, loads, stores. Domain structure (tensor algebra, dataflow graphs, hardware circuits — or geometry) is destroyed on entry. If your domain's optimizations live at a higher level, LLVM arrives too late to help. By the mid-2010s every ML framework was inventing its own incompatible graph IR above LLVM — chaos that motivated…

Module 6 · 50 min

MLIR: many IRs, one infrastructure

MLIR (Lattner again, Google, 2019; now an LLVM subproject) is the generalization: instead of one IR at one level, provide infrastructure for building many IRs ("dialects") that coexist in one module and lower progressively into each other.

  • Dialects: namespaced op vocabularies — linalg.matmul, affine.for, gpu.launch, llvm.add — each with its own verifier and semantics, all sharing one SSA/type/pass framework. Anyone can add one.
  • Progressive lowering: instead of one cliff-edge drop to LLVM IR, walk down a staircase: tensor graph → linalg → loops → vectors → llvm. Each step is a small, testable legalization pass, and optimizations run at whichever level they're natural.
  • Mixed levels in one function: during lowering, half your ops can be high-level and half low-level, simultaneously. The IR doesn't force a single abstraction level at any moment.
// two dialects coexisting mid-lowering (illustrative)
func.func @f(%t: tensor<4x8xf32>) {
  %0 = linalg.matmul ...          // still high-level tensor algebra
  %1 = arith.addf %a, %b : f32    // already scalar arithmetic
  ...
}

MLIR is why "build a compiler for your domain" stopped meaning "reimplement SSA, passes, and printers from scratch." Hardware (CIRCT), quantum, fortran (flang), ML — all build dialect stacks on it.

The transferable design lesson (Course 3 lives here): when a domain spans multiple abstraction levels, don't standardize one level and force everything through it. Let levels coexist with explicit, small lowerings between them — and keep high-level ops alive as long as possible, because every lowering is a one-way door.

Module 7 · 60 min

Relax: cross-level IR + first-class symbolic shapes

Relax: Composable Abstractions for End-to-End Dynamic Machine Learning (Lai, Shao, Feng, Lyubomirsky et al., ASPLOS 2025 — now the graph IR of Apache TVM) attacks a specific pain: compiling LLMs, whose tensor shapes are dynamic (batch size, sequence length unknown until runtime), onto dozens of targets. Its three ideas are the most transferable compiler ideas of the decade for any multi-level domain. The PDF is linked from this repo's report; here's the anatomy:

Idea 1 — Cross-level abstraction

Traditional ML compilers stack separate IRs (computation graph → loop-level tensor program → library calls) with single-shot lowering between them: once you drop a level, the level above is gone. Relax instead holds all three in one program:

def main(x: Tensor(("n", 512), "f32"), w: Tensor((512, 1536), "f32")):
    n = sym_var()
    with dataflow():
        lv0: Tensor((n,1536)) = call_tir(mm, [x, w], ...)       # loop-level tensor program
        lv1: Tensor((n,3,8,64)) = reshape(lv0, shape(n,3,8,64)) # graph-level op
        lv2: Tensor((n,8,64)) = call_dps_library(               # opaque vendor library
              "cutlass.attention", [lv1], ...)

Because graph ops, loop programs, and opaque library calls coexist, passes can do partial lowering (lower just the matmuls to cuBLAS, optimize the rest yourself), analysis feedback (inspect a loop program, learn "this is elementwise," use that to drive graph-level fusion), and cross-level transforms (lift a kernel's scratch buffer up to the graph's memory planner).

Idea 2 — First-class symbolic shapes

Older IRs mark dynamic dimensions as ?/"Any" — erasing everything. Relax keeps a symbolic expression per dimension and propagates relations: if input is (n, 4), flatten's output is (4*n,) — still tied to the same n. Erasure destroys optimization (can these two buffers be shared? unknowable with "Any"; provable with 4*n = 4*n). The general lesson: when you must ship something dynamic/unknown, carry the symbolic relation, not a baked value or a shrug.

Idea 3 — match_cast: assert structure, check at runtime

Some shapes are genuinely unknowable statically (output of unique()). Relax lets the program assert a symbolic shape — lv3 = match_cast(lv2, Tensor((m,), "f32")) — introducing new symbolic variable m, backed by a cheap runtime check that errors on violation. It's a first-class IR node (relax.expr.MatchCast, a binding in TVM's source): first occurrence of m binds it from the runtime value; later occurrences assert equality. Static when possible, checked-dynamic as the fallback.

Results context: these three ideas let one compiled artifact serve any batch size/sequence length, enable dynamic-shape memory planning (~40% activation memory reduction on Llama decode in the paper), and deploy LLMs to phones and WebGPU.

Module 8 · 60 min

The dark arts + capstone

Going backwards and sideways

  • Decompilation (binary → source): reconstruct control flow, types, names from what lowering left. Modern tools (Ghidra, Hex-Rays) produce a source, never the source. It's search/inference — increasingly ML-assisted.
  • Binary translation (ISA → ISA): run x86 binaries on ARM (Apple's Rosetta 2). Hardest part isn't instructions — it's semantic mismatches between machines (memory ordering, flags, float behavior). Sound familiar? Two geometry kernels with different tolerance regimes are two ISAs with different float semantics.
  • Transpilation (source → source, editability preserved): C++ → Rust and friends. Famously brutal, because the target must remain idiomatic and maintainable — you're translating intent, not behavior. (This — not compilation — is what "CAD interop" actually asks for. Course 3, again.)

Trust: how compilers are tested

  • Golden/round-trip tests: print → parse → print must be a fixed point (this repo's test suite does exactly this on the recipe IR and the .scad emitter).
  • Differential testing: generate random programs, run through multiple compilers/backends, compare outputs. Csmith found 400+ bugs in GCC/LLVM this way. The general recipe: N independent implementations of the same spec = a free oracle.
  • Conformance suites define what "implements the language" means. No suite, no interoperability — only vibes.

Capstone — answer cold

1. Why can't you optimize what you erased?

Optimizations prove facts; erasure (to "Any", to scalars, to triangles) removes the information the proof needs. Keep symbolic structure as long as possible — Relax's shapes, MLIR's high-level dialects.

2. State LLVM's three preconditions.

Shared abstract machine to lower to; specified semantics (incl. a UB contract); aligned incentives/funding. Remove any one and the universal-IR play fails.

3. MLIR vs LLVM in one sentence.

LLVM standardizes one low-level IR; MLIR standardizes the infrastructure for many coexisting domain-level IRs with progressive lowering between them.

4. Relax's three ideas in one line each.

(1) Graph ops, loop programs, and opaque library calls in one IR → partial lowering & cross-level optimization. (2) Dynamic dimensions as symbolic expressions, relations preserved. (3) match_cast: assert + runtime-check structure when static knowledge runs out.

5. Why is transpilation harder than compilation?

Compilation only preserves behavior; transpilation must preserve editability and idiom — the high-level structure that lowering is allowed to destroy.
