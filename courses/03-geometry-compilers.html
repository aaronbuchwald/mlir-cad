Module 1 · 40 min

Workflow 1 — export is compilation

Trace one Revit wall (hosted window, joined at both ends) through export, watching what each "lowering pass" destroys. This is a real pipeline, annotated in compiler terms:

  1. Source: wall = family type + parameters + behavioral relations (join rules, host for the window, top constrained to Level 3). Meaning defined by Revit's change-propagation engine — semantics live in the vendor runtime.
  2. Lowering 1 — behavior → relations: IFC can record the results of behavior (IfcRelConnectsPathElements, IfcRelVoidsElement + IfcRelFillsElement) but not the rules. The join logic — compiled C++ — has no representation. Compiler twin: emitting a trace of what the program did, instead of the program.
  3. Lowering 2 — parametrics → shallow recipe: the wall body can become IfcExtrudedAreaSolid (profile + direction + depth) with the opening as a boolean subtraction. Note what died here: height = Level3.elev − Level2.elev baked to 3000.0. The relation is erased, exactly like erasing a symbolic shape to "Any" (Course 2, Module 7).
  4. Lowering 3 — recipe → evaluated geometry: for anything nonstandard, exporters fall to B-rep or tessellation. And the certified exchange profile (IFC4 Reference View) is deliberately tessellation-oriented — the conformance-tested path is the lowest one. Compiler twin: the only certified output is -O0 assembly.
  5. Object code: a .ifc file. Displayable everywhere, editable nowhere.
Key asymmetry to internalize: each lowering was locally reasonable; the composition is a one-way trapdoor. No single pass "broke" the model — semantics leaked out one level at a time. Compilers accept this because binaries only need to run. Buildings need re-editing for decades — the artifact's purpose is different, so wholesale adoption of the compile-down model was a category error.

The wall's height was defined as Level3.elev − Level2.elev. After IFC export it is 3000.0. What compiler failure is this?

Module 2 · 40 min

Workflow 2 — import is decompilation

Now run the pipeline backwards: a consultant sends you that .ifc and you must continue designing in your tool. You are now a decompiler. Your options, exactly parallel to reverse engineering a binary:

  1. Wrap it (what tools actually do): Revit → DirectShape. The geometry displays, clashes, schedules — but is frozen. Equivalent: run the binary in a sandbox; don't pretend you understand it.
  2. Pattern-match idioms: recognize "vertical extrusion, 300mm profile, boolean opening" → probably a wall with a window. This is what "parametric-preferred" importers and academic semantic enrichment (rule-based, GNN-based) do. Equivalent: a decompiler recognizing loop idioms and calling conventions. Works on clean output; breaks on anything creative.
  3. Program synthesis / ML lifting: search for a recipe whose evaluation matches the target: InverseCSG (mesh → CSG program), Point2CAD (point cloud → B-rep), CAD-Recode (point cloud → executable CadQuery, via an LLM), Szalinski (e-graphs compressing flat CSG into editable parametric programs — with loops! — by equality saturation). Equivalent: modern ML-assisted decompilation. Research-grade but improving fast.

The reason all three struggle is Course 2's iron law: lifting is search over what lowering destroyed. One shape has infinitely many recipes; you want not a recipe but a maintainable, idiomatic one matching the original design intent — transpilation-hard, not compilation-hard.

Module 3 · 30 min

Workflow 3 — cross-kernel exchange is binary translation

Even with zero parametrics — pure evaluated B-rep, STEP file, "just the shape" — exchange fails routinely. The compiler lens says why, in three beats:

  1. Different machines: Parasolid (ε≈1e-8, per-entity tolerances), ACIS (ε≈1e-6), CGM, OCCT — each a self-consistent "ISA" with private float semantics. A solid watertight on machine A has real gaps on machine B.
  2. Unrepresentable instructions: kernels carry private procedural surfaces (rolling-ball blends, lazy intersection curves) that the neutral format can't express — approximated on export, shifting geometry by exactly the amounts that violate ε.
  3. The patch industry: healing middleware re-stitches, re-trims, re-tolerances — binary fixups so the target loader accepts the module. Its 30-year persistence is the proof that the "machines" genuinely differ.

And the gap nobody has filled: there is no Csmith for geometry kernels — no systematic differential-testing campaign generating random solids and cross-checking kernels against each other. N kernels are a free oracle no one has used. (Verified absent as of July 2026 — see docs/research/compilers-for-cad.md.)

Module 4 · 45 min

The fix: what MLIR + Relax prescribe

Assemble the prescription from the three workflows' failures:

  1. From MLIR — dialects, not a single level. Don't standardize one representation (that's STEP/IFC, and it's the disease). Let intent, recipe, exact geometry, and mesh coexist as dialects in one artifact with small, explicit lowerings. Consumers execute the highest dialect they speak. Degradation is per element, never per file — partial lowering, exactly as MLIR mixes dialects mid-flight.
  2. From Relax idea 2 — never bake relations. (wall_len − win_w)/2 travels as an expression. Where dynamism/unknowns force baking a value, keep the symbolic form attached beside it.
  3. From Relax idea 3 — match_cast at the exchange boundary. Ship recipe and baked oracle; the importer re-evaluates the recipe on its own kernel and diffs against the oracle. Within tolerance → element goes LIVE (parametric on the new kernel). Unsupported op or divergence → checked FALLBACK to the baked level, flagged. The tolerance problem stops being silent corruption and becomes a contract violation with a handler.
  4. From Relax idea 1 — opaque calls are first-class. Where semantics are irreducibly proprietary (Revit's join resolution), represent them as calls to named external evaluators — call_dps_library for geometry. Honest FFI beats fake portability.
  5. From LLVM's preconditions — the missing substrate. Geometry lacks specified semantics (no kernel spec) and aligned incentives (lock-in pays). Differential testing can manufacture an empirical spec; AI demand for liftable representations may supply the incentive. Until then, expect this architecture to win in wedges, not by standards decree.

Check yourself

Dialects: recipe.param/export, expr.add/sub/mul/div, geom.box/cylinder/translate/union/difference/ repeat_x/fillet. The symbolic relation: %free = expr.sub %wall_len, %win_w; %win_x = expr.div %free, 2.0. The B-rep-only op: geom.fillet — mesh kernels have no edges to fillet.

Lab 2 — run the six acts, name the compiler concept (30 min)

Run python demo.py. For each act, say aloud which Course 2 concept it demonstrates, then check:

Answer key
  1. Act 1 (OCCT bake): lowering to machine A + emitting the artifact with debug info (recipe + oracle + provenance).
  2. Act 2 (Manifold import): match_cast at a kernel boundary; per-element partial lowering — fillet → checked FALLBACK, the rest LIVE.
  3. Act 3 (12 segments): differential semantics between "ISAs" caught by the tolerance contract — the (N/2π)·sin(2π/N) factor is the two machines' float disagreement, measured.
  4. Act 4 (edit win_w, col_n post-exchange): proof the program stayed source, not object code — regeneration on the foreign kernel. The frozen pedestal shows the IFC condition, per-element instead of per-file.
  5. Act 5 (emit + lift .scad): transpilation to another system's source language with intent preserved (live params, expressions, a for-loop) + source-level lifting; the import() line is honest FFI to baked geometry.
  6. Act 6: the scoreboard — read it as a compiler pipeline diagram.
Lab 3 — break the contract (20 min)

In demo.py, change EPS = 0.005 to 0.0005 (0.05%) and rerun. Predict first: what happens to the colonnade at 64 segments, whose faceting deficit is −0.16%?

Expected result

Colonnade flips LIVE → DIVERGED (0.16% > 0.05%): the same two kernels, the same recipe — only the contract changed. Boxes stay LIVE (mesh boxes are exact). Lesson: the tolerance contract is a negotiated engineering parameter, not physics — exactly like choosing float equality epsilons or -ffast-math budgets. Restore EPS afterwards.

Lab 4 — extend the dialect (60–90 min)

Add geom.rotate_z %solid, angle_degrees end-to-end, following the checklist in CLAUDE.md (Conventions): OP_SPECS entry in geomir/ir.py → evaluator case in eval.py → all three backends (OCCT: rotate about the z-axis through the origin; Manifold: .rotate([0,0,deg]); sampler: inverse-rotate the query points — mind the bbox: rotate its 8 corners and rebound) → scad.py emit (rotate([0,0,a])) and lift → a test with a closed-form check (hint: rotation preserves volume — assert the wall's volume is unchanged at any angle, and its bbox changes). Run python tests/run_tests.py until green.

Design questions to answer before coding

Is rotate_z representable in all backends (yes — so no fallback path needed, unlike fillet)? Does the scad lifter grammar need a new statement form (yes — mirror how translate is parsed)? Does the emitted form stay in the liftable subset (keep it canonical: rotate([0, 0, expr]) exactly)?

Lab 5 — hand-edit and lift (15 min)

Open out/model.scad in a text editor (or OpenSCAD — drag the customizer sliders). Change win_w = 1500; to 2000 and add a comment. Then in Python: from geomir import lift_scad; m,w,_ = lift_scad(open("out/model.scad").read()); print(m.params()["win_w"], w). Module 6 · 30 min

Frontier: the three wedges + capstone

Where a person with both courses in their head could actually move the field (argued in full in docs/aec-geometry-ir-analysis.md §8–9):

  1. AI verification infrastructure. LLMs generate code-CAD (CadQuery, KCL); someone must verify, canonicalize, and diff their output against intent. An IR with verifiers and multi-kernel lowering is that substrate — demand exists today, independent of any standards body.
  2. Lifting as a product. "Dumb STEP/IFC in, native editable parametric out" — what Elysium CADfeature and ITI Proficiency sell at enterprise prices on old technology. Modern stack: ML lifting (CAD-Recode-line) + e-graph canonicalization to the target's idiom (Szalinski-line) + re-evaluation checking (match_cast) for trust.
  3. Differential kernel testing. Csmith-for-geometry. Cheapest to start, publishable, and it manufactures the empirical semantics spec that any future geometry-IR standard would need. The confirmed gap is sitting there.

And the sober note that keeps you honest: LLVM had aligned incentives; CAD's incumbent profits from the lossiness. Expect the architecture to arrive via AI-era wedges, not committee decree — watch whether IFC5's component model ever grows a recipe dialect and symbolic expressions.

Capstone — the one-breath versions

1. State the whole thesis in three sentences.

CAD interop fails because tools exchange evaluated geometry (object code) while intent lives in proprietary runtimes, and even the object code targets kernels with different numeric semantics. The fix is not one shared representation but an MLIR-style multi-level artifact — intent, recipe, exact geometry, mesh — with symbolic relations never baked and Relax-style match_cast validation at every kernel boundary, degrading per element. The blockers are unspecified kernel semantics and incentives, not IR design.

2. Why did the demo need TWO kernels?

One kernel proves nothing about exchange — the entire problem is semantic disagreement between evaluators. OCCT (analytic B-rep) vs Manifold (polyhedral mesh) supplies real, measurable divergence ((N/2π)·sin(2π/N)) and a real capability gap (fillet), so the contract and fallback have something genuine to catch.

3. Where exactly do LLVM, MLIR, and Relax each enter the story?

LLVM: the economic template (M+N via a shared IR) and the three preconditions that geometry fails — which explains 30 years of failed neutral formats. MLIR: the architectural correction — coexisting dialects, progressive per-element lowering, no single sacred level. Relax: the operational mechanics — cross-level artifacts with opaque foreign calls, symbolic relations instead of baked values, and match_cast's assert-then-check pattern at the boundary.
